Build Task #15 — Plan Import Receipt + Undo
What and why
After a user clicks Apply in the Plan Importer, the current success state just shows a count (“X events, Y tasks, Z placements applied”) and clears. There’s no way to see what actually landed where, or to reverse a bad placement without hunting through the app manually. This task adds a receipt screen that appears after every apply, plus per-item undo for placements.
Done looks like
After Apply completes, instead of just a toast, the results area transitions to a receipt view showing three collapsible sections — Events Added, Tasks Added, Placements Filed. Each item in the receipt shows what was created or updated and where. Placements show the destination label, the match_hint, the field, and a one-line summary of the value (truncated to ~80 chars). Each placement row has an Undo button. Events and tasks show title/date/person with no undo (destructive undo for those is out of scope for v1). A “Done” button at the bottom clears the importer back to the paste phase.
Clicking Undo on a placement sends a POST to a new route /plan-import-undo-placement with enough information to reverse the write: destination, field, match_hint, the value that was appended/written, and the action that was taken (UPDATE/APPEND/CREATE). The server reverses it: appended strings are stripped, appended list items are removed, created records (intentions, thank-yous) are deleted by id, notes-inbox fallbacks are deleted by id. On success the placement row is marked “↩ Undone” and the Undo button is disabled.
The receipt is built from the response that /plan-import-apply already returns — extend that response to include a receipt array alongside events_added, tasks_added, placements_applied. Each receipt item carries: type (event/task/placement), label (human-readable destination), title (event title, task text, or placement match_hint), field (placements only), value_preview (first 80 chars of value, placements only), action, and undo_id (a uuid the server generates per placement at apply time, stored temporarily in a small in-memory dict so the undo route can look up what to reverse).
Out of scope
Undo for events and tasks. Session recovery, history, companion panel — untouched. Persistent undo across server restarts (the in-memory dict is enough; if the server restarts the Undo buttons just silently disable).
Steps
	1.	In /plan-import-apply in app.py, after each placement is processed, generate a undo_id = uuid4().hex[:12], store the reversal metadata in a module-level dict _placement_undo_registry keyed by undo_id, and include a receipt entry in the response. The reversal metadata needs: destination, field, match_hint, action, value_written, record_id (for CREATE actions where a new record was made). Cap the registry at 50 entries (evict oldest) so it doesn’t grow unbounded.
	2.	Add a new POST route /plan-import-undo-placement in app.py. It receives {undo_id}, looks it up in _placement_undo_registry, performs the reverse operation using the same dispatch logic as the apply handler (mirror image: APPEND → strip the appended value, CREATE → delete the record, UPDATE → restore previous value if stored). Returns {ok: true} or {ok: false, reason: "..."}. Remove the entry from the registry after use.
	3.	In render_plan_importer.py, add a #phase-receipt div (hidden by default, same phase-swap pattern as #phase-paste / #phase-thinking / #phase-results). After a successful apply response, hide #phase-results, show #phase-receipt, and populate it from result.receipt. Render three collapsible sections (Events Added, Tasks Added, Placements Filed) using the same .item-row card pattern already in the page. For placements, add an Undo button per row that calls a JS undoPlacement(undo_id, rowId) function. That function POSTs to /plan-import-undo-placement, and on success marks the row with “↩ Undone” and disables the button. Add a “Done” button that calls the existing resetImporter() (or equivalent clear function) to return to the paste phase.
	4.	No new files. No changes to the history system, session recovery, companion panel, or any route other than /plan-import-apply (extended response only) and the new /plan-import-undo-placement.
