Task #17 — Companion Responses Generate Actionable Items
What and why
When a companion responds during plan import consultation, their advice is currently plain text with no path into the app. If Lucy says “you should add a prep task for that drill” Lauren has to manually create it. This task makes companion responses parseable — after each response streams in, a second lightweight AI call extracts any suggested tasks or events and renders them as checkable cards below the companion’s text, using the same approve/edit/apply pattern already in the page.
Done looks like
After a companion response finishes streaming, a small secondary parse call runs against that response text. If it finds any suggested tasks or events, they appear as cards in a new “Suggested by [Companion]” strip directly below the companion’s chat bubble. Cards use the existing .item-row pattern with checkboxes. Checked cards are included when the user clicks Apply alongside the main events/tasks/placements. If the companion response contains no extractable items the strip simply doesn’t appear — no UI change for pure conversational responses.
The group council works the same way — after parseCouncilResponse() finishes coloring the sections, the same extraction runs on the full council text and surfaces a combined suggestions strip.
Out of scope
Changing the streaming behavior, the companion system prompts, the history persistence, or any companion page outside plan import. No changes to /plan-import-consult or /plan-import-group-consult server routes. The secondary parse is client-side only — a direct Anthropic API call from the browser using the existing key from app settings.
Steps
	1.	Add a client-side extraction function extractCompanionSuggestions(companionName, responseText) in render_plan_importer.py. This function calls the Anthropic API directly from the browser (same pattern already used elsewhere in the page if any, otherwise use fetch to /api/extract-suggestions — see step 2). It sends a compact prompt: “You are extracting actionable items from a planning assistant’s advice. Return only valid JSON with keys tasks and events, same schema as the plan importer. If no actionable items exist return {tasks:[], events:[]}. Text: [responseText]”. Parse the response through the existing _repair_and_parse_json equivalent in JS (the page already has JSON repair logic — use it).
	2.	Add a thin server route /api/extract-suggestions in app.py. Accepts POST with {text, companion}. Calls Anthropic with the extraction prompt above, max_tokens 1000, no streaming. Returns parsed JSON {tasks:[], events:[]}. Add to _JSON_PATHS. This keeps the API key server-side rather than exposing it in the browser.
	3.	In runConsultMessage(), after the stream finishes and consultHistories[key].push(...) runs, call extractCompanionSuggestions(key, text). On response, if tasks or events are non-empty, render a <div class="companion-suggestions"> strip below aiEl with a header “💡 Suggested by [CompanionName]” and cards built by the existing renderTaskItem() and renderEventItem() functions. Assign these items IDs prefixed cs- to avoid collisions with main plan items.
	4.	In conveneCouncil(), after parseCouncilResponse(fullText) runs, do the same extraction call on fullText and render a single combined strip labeled “💡 Council Suggestions”.
	5.	In collectApproved(), after collecting main tasks and events, also walk any .companion-suggestions .item-row elements and collect checked ones into the existing tasks/events arrays. No new payload key needed — they merge with the main items.
	6.	Add minimal CSS for .companion-suggestions — a left border in the companion’s accent color, slightly inset, visually connected to the chat bubble above it.
Constraints
Edit only app.py (new route only) and render_plan_importer.py. No changes to any companion render file, history file, or existing route. If render_plan_importer.py would exceed 2000 lines as a result, flag before proceeding.
